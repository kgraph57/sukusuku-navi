# すくすくナビ 事業計画書

港区の子育て、もう迷わない。

2026年2月｜愛育病院 小児科 岡本賢

------------------------------------------------------------------------

## 1. エグゼクティブサマリー

すくすくナビは、港区の小児科医が開発する子育て支援Webアプリ。医師監修の医療情報、港区の行政手続き・給付金情報、予防接種スケジュール、受診判断トリアージを一つに統合し、**「生年月日を入れるだけで、今週やるべきことがわかる」**パーソナライズドサービスを提供する。

**核となる価値**: 医療信頼性（現役小児科医監修）× 行政アクション（港区特化の手続き案内）× 地域密着（かかりつけ医としての継続関係）

**目標**: 2026年内に港区の子育て世帯の認知率30%、アクティブユーザー500人を達成し、社会実装の実績を構築する。

## 2. 解決する課題

### 2.1 課題の構造

    保護者の不安
      ├── 医療情報の信頼性問題（ネット検索→不確かな情報→不安増大）
      ├── 行政手続きの複雑さ（何を・いつ・どこで申請すべきかわからない）
      ├── 予防接種の管理困難（種類が多い・スケジュールが複雑）
      └── 受診判断の孤立（夜中に子どもが発熱→誰にも聞けない）

### 2.2 既存サービスの限界

| サービス | 強み         | 欠けているもの               |
|----------|--------------|------------------------------|
| ぴよログ | 育児記録     | 医療情報、行政手続き         |
| 母子モ   | 自治体連携   | 医師監修コンテンツ、受診判断 |
| ママリ   | コミュニティ | エビデンス、地域特化         |
| 港区HP   | 公式情報     | わかりやすさ、パーソナライズ |

**すくすくナビの独自ポジション**: 上記すべてを「一人の医師の顔が見える形で」統合する唯一のサービス。

## 3. プロダクト概要

### 3.1 実装済み機能（2026年2月時点）

| 機能 | 概要 | 完成度 |
|----|----|----|
| 医療記事 | おかもん先生監修60記事（Q&A形式・参考文献付き） | 95% |
| 給付金シミュレーター | 家族構成・収入から受給可能な制度を自動判定 | 90% |
| 受診判断トリアージ | 症状入力→「今すぐ受診/明日受診/自宅ケア」を判定 | 85% |
| 小児科マップ | 港区内12+小児科のインタラクティブ地図 | 90% |
| 手続きチェックリスト | 出生届から各種申請までの進捗管理 | 80% |
| 保育園検索 | 港区内80+認可・認証保育園の一覧・フィルタ | 90% |
| 健診ガイド | 乳幼児健診（1ヶ月〜5歳）の時期・内容・準備 | 85% |
| 予防接種情報 | 20+ワクチンの詳細情報・スケジュール表 | 80% |
| マイページ | お子さん登録・タイムライン・接種記録・マイルストーン | 60% |

### 3.2 技術スタック

- **フロントエンド**: Next.js 16 + React 19 + TypeScript + Tailwind CSS v4
- **コンテンツ**: MDX（医療記事）+ JSON（構造化データ）
- **認証**: Supabase Auth（ログインなしでもlocalStorageで利用可）
- **地図**: Leaflet + React Leaflet
- **分析**: PostHog
- **デプロイ**: Vercel（GitHub Actions自動デプロイ）
- **PWA**: Service Worker対応（インストール可能）

## 7. 競合優位性と参入障壁

### 7.1 なぜ「おかもん先生」でなければダメなのか

1.  **医師の実名・顔出し**: 匿名の医療情報サイトとは信頼性が根本的に異なる
2.  **愛育病院という配布チャネル**: 自分の勤務先＝最大のユーザー獲得拠点
3.  **診察室での継続的接点**: オンラインとオフラインの信頼が相互強化
4.  **コンテンツの質**: 全記事に査読論文レベルの参考文献付き
5.  **技術力**: 医師自身が開発可能（外注コストゼロ、迅速な改善）

### 7.2 参入障壁

- 「医師×エンジニア×港区勤務」の組み合わせは再現困難
- 60記事分のコンテンツ蓄積は後発が追いつきにくい
- 地域密着の信頼関係は広告では買えない

## 8. 収益モデル（将来構想）

**現時点の方針: マネタイズよりも社会実装を優先**

サービスは完全無料で提供し、社会インパクトの最大化に集中する。将来的な持続可能性のオプションとして以下を検討：

| モデル     | 概要                                         | 時期           |
|------------|----------------------------------------------|----------------|
| 自治体契約 | ホワイトラベル提供の対価として自治体から受託 | 2027年〜       |
| 研究助成金 | 厚労科研費、JST等の研究費獲得                | 2026年〜応募可 |
| 企業協賛   | 地域企業（薬局、ベビー用品店等）のスポンサー | 将来検討       |

**重要な原則**: 広告モデルは採用しない。医療情報の信頼性と利益相反の回避を最優先する。

## 9. リスクと対策

| リスク | 影響度 | 対策 |
|----|----|----|
| ユーザーが集まらない | 高 | 愛育病院チャネルで確実に初期ユーザーを獲得。10人→50人→200人と段階的に |
| 医療情報の誤り・訴訟リスク | 高 | 全記事に免責事項明記。個別医療相談との線引き。参考文献の厳格な管理 |
| 開発者の時間不足 | 中 | MVPに集中。「今週やること」の1機能を完璧にする。他は後回し |
| 行政連携が進まない | 中 | 連携なしでもサービスは成立。実績を積んでから再提案 |
| 競合の参入 | 低 | 医師×エンジニア×地域密着の組み合わせは容易に再現不可 |
| 個人情報漏洩 | 高 | ログインなしでも利用可能な設計。Supabase RLS。最小限のデータ収集 |

## 付録B: ホワイトラベル展開の技術設計

すくすくナビは設計段階から他自治体への展開を想定：

- **データ層の分離**: 行政情報はすべてJSON（data/ディレクトリ）で管理
  - programs.json: 助成金・制度情報
  - clinics.json: 小児科クリニック
  - nurseries.json: 保育園
  - checkups.json: 乳幼児健診
- **コンテンツ層の共通化**: 医療記事（MDX）は自治体非依存
- **UIの共通化**: コンポーネントは自治体名をprops化可能

**展開に必要な作業**: JSONデータの差し替え + 自治体ロゴ・色の変更のみ。
